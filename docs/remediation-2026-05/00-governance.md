# AntTrader 工程治理与硬性约束

## 1. 本文档地位

本文档定义 AntTrader 项目整改期间的硬性工程规则。

所有新代码、新接口、新文档、新重构必须遵守本文档。若历史实现与本文档冲突，以本文档作为整改目标。

## 2. 前后端职责边界

### 2.1 前端职责

前端只负责：

- 页面布局。
- 表单输入。
- 用户交互。
- 图表渲染。
- 国际化展示。
- ConnectRPC client 调用封装。
- 轻量 UI 状态，例如弹窗开关、当前 tab、筛选条件草稿。

前端不得负责：

- 交易盈亏计算。
- 风险指标计算。
- 账户权益聚合。
- 回测指标计算。
- 策略信号计算。
- AI 结果裁决。
- 权限最终判断。
- 风控决策。
- 订单有效性最终判断。
- 跨账户统计聚合。

### 2.2 后端职责

后端负责所有业务计算与系统决策：

- 用户权限与 API Key scope 判断。
- 账户状态、交易权限、连接状态计算。
- 持仓、订单、历史、权益、风险指标聚合。
- 手动交易与自动交易风控。
- 策略模板校验、发布状态机、调度状态机。
- 回测任务编排、结果存储、风险评分。
- AI provider 选择、Agent 编排、Debate 状态推进。
- 日志、审计、指标、运维健康状态。

### 2.3 策略服务职责

策略服务属于后端计算体系的一部分，不属于前端能力延伸。

策略服务负责：

- Python 策略代码校验。
- 策略沙箱执行。
- 回测引擎。
- 指标计算。
- 策略记忆与客观评分。

策略服务不得直接暴露给浏览器作为业务 API。

## 3. 通信协议硬约束

### 3.1 业务通信统一 ConnectRPC

浏览器与后端之间的业务接口必须使用 ConnectRPC。

禁止新增：

- 新的 `/api/...` 业务 REST 接口。
- 前端直接 `fetch('/api/...')` 调用业务接口。
- 前端直接调用策略服务接口。
- 绕过 proto 的宽松 JSON 业务接口。

### 3.2 REST 例外

仅允许以下非业务 REST 例外：

- `/health`、`/health/live`、`/health/ready`、`/health/deps`。
- 静态资源服务。
- 临时迁移期兼容端点，但必须登记在迁移清单中，并给出删除目标版本。

### 3.3 内部服务通信

目标状态：内部服务通信也应尽量收敛到明确契约。

整改期允许后端通过内部 client 调用策略服务现有 HTTP 接口，但必须满足：

- 浏览器不可直接访问策略服务。
- Go 后端必须封装为 `PythonStrategyService` 或后续 `StrategyRuntimeService`。
- 对外业务语义必须在 proto 中表达。
- 迁移计划中必须记录这些内部 HTTP 调用的最终去向。

## 4. 单文件 800 行硬约束

除 Markdown 文档外，所有文件必须满足：

```text
单文件行数 <= 800
```

约束适用范围包括但不限于：

- Go 源码。
- TypeScript / TSX 源码。
- Python 源码。
- Shell 脚本。
- SQL 迁移脚本。
- YAML / JSON 配置。
- proto 文件。
- proto 自动生成的 Go / TS 文件。
- 其他构建生成代码。

### 4.1 超限处理原则

超过 800 行时必须拆分：

- 大 service 拆成按领域或方法组文件。
- 大 React 页面拆成 container、components、hooks、client、types。
- 大 Python 模块拆成 router、schema、service、engine。
- 大 proto 拆成多个领域 proto。
- 生成代码超限时，必须拆 proto，而不是忽略生成物。

### 4.2 生成代码特别要求

proto 自动生成代码不得例外。

如果生成文件超过 800 行，整改方向是：

- 拆分 proto service。
- 拆分 message 定义。
- 按领域拆包。
- 控制单个 service 的 RPC 数量。
- 控制单个 message 的复杂度。
- 调整生成输出目录，使生成文件自然分散。

### 4.3 外部协议契约例外

`mtproto/mt4.proto` 与 `mtproto/mt5.proto` 来自外部上游协议契约，不属于项目自有业务 proto。

处理原则：

- 不拆分、不改写这两个外部 proto。
- 不将其纳入项目自有 proto 拆分整改范围。
- 如果需要调整 MT4/MT5 生成代码结构，应优先通过隔离外部生成物或独立模块边界处理，不能反向修改外部协议契约。

## 5. 文档规则

Markdown 文档不受 800 行限制，但必须遵守：

- 标明当前状态还是目标设计。
- 标明是否已实现。
- 标明迁移阶段。
- 避免把占位接口描述成已完成能力。
- 废弃文档不得直接删除，先归档或建立替代表。

## 6. 评审检查项

每次代码评审必须检查：

- 是否新增 REST 业务接口。
- 是否把计算放到了前端。
- 是否有文件超过 800 行。
- 是否有 proto 生成文件超过 800 行。
- 是否修改 proto 后重新生成并检查产物。
- 是否有质量检查记录。
- 是否需要容器重建和重启。

## 7. 整改优先级

优先整改顺序：

1. 前端业务 REST 调用。
2. 后端 `/api/...` 业务 REST handler。
3. 前端计算逻辑。
4. 超 800 行源码文件。
5. 超 800 行 proto 与生成代码。
6. 旧文档误导性描述。
