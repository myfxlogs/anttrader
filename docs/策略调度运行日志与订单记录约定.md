# 策略调度运行日志与订单记录约定

本文约束 **策略调度（Strategy Schedule v2）** 在「运行记录 / 交易记录」页应展示的数据来源与写入时机，与实现代码一致。

## 运行记录（执行日志）

合并查询（见 `LogRepository.GetScheduleRunLogs`）包含：

1. **`strategy_execution_logs`**：`schedule_id` 匹配的所有行（含市价/挂单信号类型字符串，如 `buy`、`buy_limit`、`close` 等）。
2. **`system_operation_logs`**：`module = 'strategy_schedule'` 且 `resource_type = 'schedule'`、`resource_id = schedule_id`（如启停调度）。

调度在 **CLOSE** 信号下通过网关平仓时，必须写入与开仓对称的 **`strategy_execution_logs`**（`signal_type = close`），以便运行记录中出现「平仓」类执行行。底层 `schedule_v2_close` 操作日志仍为 `module=execution`，不进入上述 UNION；以执行表为准。

## 交易记录（订单历史）

列表为 **`order_history`** 中 `schedule_id` 匹配的行。

- **开仓 / 挂单**：`OrderSend` 成功后插入一行（`close_time` 为空直至平仓）。
- **平仓**：`OrderClose` 成功后对**同一 `ticket` + schedule_id** 的行执行 `UPDATE`，写入 `close_price`、`close_time`、`profit`、`swap`、`commission`。若不存在对应开仓行（例如仓位非本调度写入），则插入一条带 `schedule_id` 的补录行（含开仓价来自持仓快照、平仓价来自成交回报）。

## 信号到订单类型

Python 返回的 `signal` 经 `scheduleOrderTypeFromSignal` 规范为小写引擎类型（如 `BUY`→`buy`，`BUY_LIMIT` / `buy_limit`→`buy_limit`），再交给 `OrderSend` / 风控默认值逻辑。

## 变更记录

- 引入平仓执行日志与 `order_history` 更新/补录后，调度日志页应能同时看到开仓、挂单与平仓相关记录。
