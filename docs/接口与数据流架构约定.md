# 接口与数据流架构约定

> **本文档是代码实施的依据与约束。**  
> 凡涉及对外 API、实时推送、长耗时任务、多 Agent 编排等改动，须先修订本文档（或本文档引用的子文档）再改代码；未纳入文档的设计不得作为合并依据。

## 1. 文档驱动原则

1. **`docs/` 目录**下、以**中文文件名**保存的架构/接口/流程说明，与本文件共同构成「可实施规范」。
2. **实施顺序**：分析 → **更新文档**（本文件或关联文档）→ 再编写/修改代码。  
3. 若实现中发现规范不可行，应**先改文档**（写明原因、边界、例外），再改代码，禁止「先写代码再补文档」式扩大范围。

## 2. 允许的技术栈（对外与对内服务间）

| 能力 | 首选技术 | 说明 |
|------|----------|------|
| 浏览器/网关可调用的 RPC | **Connect RPC**（含 Unary 与 **流式 RPC**） | 与现有 `proto`、`@connectrpc/connect-web` 一致； Unary 可走 HTTP/1.1，流式须注意端到端 HTTP/2 与代理配置。 |
| 服务间高性能调用 | **gRPC**（原生） | 与 Connect 共用 Protobuf 契约时保持 `.proto` 单一事实来源；具体路由以仓库既有分层为准。 |
| 浏览器单向实时推送（含异步 Job 进度） | **SSE**（`text/event-stream`） | 事件驱动：服务端在状态变化时推送；客户端用 `EventSource` 或等价 API 消费。**不作为** WebSocket 的替代双向通道；需双向控制时仍通过 **Connect 流式 RPC** 或 Unary 完成。为兼容 `EventSource`，允许对**单一** SSE URL 使用 **`GET`**（仅流式订阅语义），**不视为** REST CRUD 资源面。 |

---

## 3. 明确拒绝的形态

| 形态 | 结论 | 说明 |
|------|------|------|
| **WebSocket** | **禁止新引入** | 避免与 HTTP/2、企业代理、`Upgrade` 头冲突；双向需求由 **Connect 流式** 或 **Unary + SSE** 组合满足。 |
| **REST 作为新 API 面** | **禁止** | 不新增以「资源动词 + JSON CRUD」为主的对外 REST 路由作为**业务能力入口**；存量 REST 仅允许**窄幅缺陷修复**，新建能力须 **Connect / gRPC / SSE**。 |
| **轮询作为默认模式** | **禁止** | 不得以「定时 `GET` 拉状态」作为实时性的主路径。 |
| **定时任务作为默认编排** | **禁止** | 不得以 crontab 式「周期性扫库/扫接口」替代**事件订阅、队列消费、流式推送**。 |

---

## 4. 数据流优先（实施约束）

### 4.1 优先级（从高到低）

1. **Connect 流式 RPC**（server / client / bidi，以 `.proto` 定义为准）  
2. **SSE**：适用于仅服务端→浏览器推送、且希望与 RPC 路径分离的长任务进度、通知类场景  
3. **gRPC 流**（服务间）  
4. **Connect Unary / gRPC Unary**：请求-响应明确且无需持续推送时使用  

### 4.2 长耗时与边缘超时（与 524 治本一致）

- 长耗时工作流（如辩论 **agent kickoff**、**代码生成**）须采用 **异步 Job**：首包快速返回（如 `job_id`），进度与结果通过 **SSE** 或 **Connect server-stream** 推送，**不得**依赖「单条 Unary 挂到 LLM 结束」穿越边缘网关。  
- 参考：`docs/边缘网关与长连接问题处理参考.md` 中 Cloudflare 524 与异步化结论。

### 4.3 例外：允许轮询或定时的条件

仅在同时满足下列条件下，允许在实现中使用**轮询**或**定时任务**，且须在 PR/文档中**逐条写明**：

1. **客观上无法用数据流实现**（例如：第三方仅提供轮询 API、或目标环境禁止长连接且 SSE 亦不可达）；  
2. 已在本文档或专题设计文档中增加**「例外条款」**小节，说明范围、频率上限、退出条件；  
3. 已评估资源与延迟成本，并优先考虑 **退避（backoff）**、**单次补拉** 而非固定高频轮询。

### 4.4 Job：**SSE 事件流为主**

| 层级 | 约定 |
|------|------|
| **主路径** | 浏览器在 Connect Unary 取得 `job_id` 后，使用 **`EventSource`** 订阅 **`GET` + `text/event-stream`** 的 Job 事件流（事件驱动）。**禁止**使用 WebSocket 承载同一语义。 |
| **断线恢复** | **退避重连同一 SSE URL**；必要时 `Last-Event-ID` / 序号重放（以 **`docs/辩论与代码生成异步任务设计.md`** 为准）。**禁止**用 Unary **周期轮询**替代 `chunk` 事件或充当流式正文的主路径。 |
| **终态对齐** | 仅在 SSE 路径经多次重连仍无法确认结果时，允许 **单次** Unary 读取 Job **终态**（非轮询循环）；须在 **`docs/辩论与代码生成异步任务设计.md` §12** 登记环境与超时策略。 |
| **实施细节** | 辩论 advance / chat 的 RPC 名、路径、事件 JSON 以 **`docs/辩论与代码生成异步任务设计.md`** 为单一实施依据；代码不得偏离该文档。 |

## 5. 与可参考开源实践的对齐（非强制拷贝代码）

- **QuantDinger**：Agent Gateway、审计、**Job + SSE** 推送长任务——对齐「异步 Job + 事件推送」，**不**把轮询当主路径。  
- **Hermes / OpenClaw**：网关、工具/MCP、子任务——对齐「常驻连接与工具边界」，仍以 **Connect + SSE** 落地，而非引入 WS。
