# 边缘网关与长连接问题处理参考（GitHub / 官方结论摘要）

本文汇总 **Connect-RPC 官方文档**、**connect-go 等仓库 Issue** 以及 **Cloudflare 524** 的权威说明，用于指导「让 Agent / 长 RPC / 流式订阅真正可用」的架构与运维决策，而非仅改前端提示文案。

---

## 1. Connect：流式与反向代理（GitHub / FAQ）

### 1.1 官方 FAQ（必读）

- **浏览器里可靠地调 server-streaming**：与整条链路上的超时、代理能力高度相关；建议在基础设施上放宽与流持续时间匹配的超时，并在客户端用较短 deadline + 重试。  
  来源：[Connect FAQ — How do I reliably call a server streaming RPC from a web browser?](https://connectrpc.com/docs/faq/)

- **Nginx 代理 Connect**：**Unary** 可走 Connect 协议且**不要求端到端 HTTP/2**；**Streaming** 通常需要 **端到端 HTTP/2**，而 FAQ 明确写：**Nginx 对这类场景支持不完整**，更推荐 **Envoy、Apache，或 HAProxy 等 TCP 层负载均衡**。  
  来源：[Connect FAQ — How do I proxy the Connect protocol through NGINX?](https://connectrpc.com/docs/faq/)

- **Go `http.Server` 的 `ReadTimeout` / `WriteTimeout`**：会作用于**整次** RPC（含流），过长流会被服务端掐断；需按部署文档区分 `ReadHeaderTimeout` 等。  
  来源：[Connect FAQ — stream error after X seconds](https://connectrpc.com/docs/faq/)

### 1.2 Go 流式实现注意点

- 文档强调：链路上每一跳代理都需良好支持 **HTTP/2**；流式往往要把代理超时调大。  
  来源：[Connect for Go — Streaming](https://connectrpc.com/docs/go/streaming/)

### 1.3 GitHub Issue（与 `ERR_HTTP2_PROTOCOL_ERROR`、Nginx 强相关）

- **[connectrpc/connect-go#760](https://github.com/connectrpc/connect-go/issues/760)**  
  现象含 `net::ERR_HTTP2_PROTOCOL_ERROR 200 (OK)`、Nginx/OpenResty 前置、同时存在 WebSocket 的 `Upgrade` / `Connection: upgrade` 头。维护者指出：**HTTP/2 与 WebSocket 的 upgrade 头组合在浏览器侧常不兼容**，建议对流与 WS **拆分路径或配置**，或对相关 location **改用 HTTP/1.1** 等策略验证。

- **[connectrpc/connect-go#879](https://github.com/connectrpc/connect-go/issues/879)**  
  讨论为流式单独设置写超时、`http.ResponseController` 等（与 Go 版本、服务端细粒度超时有关）。

- **[connectrpc/connect-go#604](https://github.com/connectrpc/connect-go/issues/604)**  
  讨论慢客户端、长驻流与**按端点配置超时**的理想形态（社区与维护者对 per-RPC 超时的设计讨论）。

**对 AntTrader 的落地启示**

1. **生产环境流式入口（`StreamService.SubscribeEvents` 等）**：优先评估 **Envoy / Caddy** 或直连可 HTTP/2 的网关，避免「Nginx 对上游仅 HTTP/1.1」的经典坑。  
2. **同域同时承载 WS 与 Connect 流**：对照 #760 检查是否对 gRPC/Connect 路径错误设置了 `Upgrade`。  
3. **应用进程**：检查 `ReadTimeout`/`WriteTimeout` 是否过短导致流被服务端主动断开。

## 2. Cloudflare HTTP 524（长耗时 Unary，如 `AdvanceDebateV2`）

### 2.1 官方定义

- **524**：Cloudflare 已连上源站，但在默认 **Proxy Read Timeout（文档写默认 120 秒量级）** 内未收到完整 HTTP 响应。长查询、大推理、过载源站均会触发。  
  来源：[Cloudflare — Error 524: a timeout occurred](https://developers.cloudflare.com/support/troubleshooting/http-status-codes/cloudflare-5xx-errors/error-524/)

- 官方建议方向包括：**缩短同步请求耗时**、对大型处理做 **状态轮询 / 异步任务**、对**必须超长**的接口使用 **Enterprise 调大 proxy_read_timeout**，或把该子域设为 **DNS Only（灰云）** 绕过代理（需权衡暴露源 IP 与安全）。

### 2.2 GitHub 上的共性结论（示例）

- 社区项目里对 524 的排查常归结为：**同步 HTTP 超过边缘允许时长**，解决思路是 **异步化、批处理变小、或绕开边缘限制**。  
  示例：[clearlydefined/service#749](https://github.com/clearlydefined/service/issues/749)（讨论 API 过慢与 Cloudflare 524 的关系）。

- **Connect 长流（如 `StreamService.SubscribeEvents`）**：浏览器侧为 **长时间挂起的 POST**；若在**边缘读超时**内没有足够字节回传（例如行情静默、或隧道/代理比源站更短），同样会出现 **524**，表现即「流被中间层掐断」——与辩论页的 **SSE（GET）** 不是同一条连接。前端已把 **524** 等归入传输类失败并 **退避重连**（见 `frontend/src/utils/streamErrors.ts`）；根本缓解仍依赖 **调大边缘与 Nginx 读超时**、或 **DNS Only / 直连子域** 承载长流。

**对 AntTrader 的落地启示**

1. **治本（推荐）**：将「可能超过 100～120 秒」的 **代码生成 / Advance** 从「单请求挂到结束」改为 **202 + job_id + 轮询 / SSE / 流式进度**，或独立 **api-direct** 子域仅给可信客户端走 DNS Only。  
2. **治标（运维）**：在具备 **Cloudflare Enterprise** 的前提下按官方 API 调整 `proxy_read_timeout`；并保证源站 Nginx 到应用的 `proxy_read_timeout` 大于边缘与 LLM 最坏耗时。  
3. **产品层**：前端重试与 `getSession` 同步状态（已实现）只能缓解失败体验，**不能替代**上述架构或网关调整。

## 3. 建议实施顺序（团队 checklist）

| 优先级 | 动作 | 目的 |
|--------|------|------|
| P0 | 确认 `t.myfxlogs.org` 前链路：是否经 Cloudflare / Nginx；524 与 H2 错误分别卡在哪一跳 | 对齐根因 |
| P0 | 长耗时 RPC 异步化或独立直连子域 | 消除 524 硬限制 |
| P1 | 流式 RPC 使用 Envoy/Caddy 或修正 Nginx H2/Upgrade | 减少 `ERR_HTTP2_PROTOCOL_ERROR` |
| P1 | 审查 Go `http.Server` 与上游代理全套 timeout | 避免服务端提前断流 |
| P2 | 客户端：短 deadline + 退避重试（流与 unary 分别策略） | 韧性 |
